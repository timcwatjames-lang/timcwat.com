1. What Is Web Accessibility?

Web accessibility (a11y) means designing and developing websites that people with disabilities can use effectively. This includes visual, auditory, motor, and cognitive disabilities.

〈 The Web is for Everyone
<!-- Without accessibility -->
<img src="chart.png">
<div onclick="submit()">Go</div>
<input type="text" placeholder="Name">

<!-- With accessibility -->
<img src="chart.png" alt="Sales chart showing 20% growth in Q3">
<button onclick="submit()">Go</button>
<label for="name">Name</label>
<input type="text" id="name">
▶ Key Principles
Perceivable — Content must be presentable to senses (alt text, captions)
Operable — Interface must work with keyboard, voice, switch
Understandable — Content and UI must be clear and predictable
Robust — Content must work with current and future technologies
💡 The a11y number: "a11y" stands for "accessibility" — 11 letters between the "a" and the "y". About 15% of the world's population has some form of disability.

2. Semantic HTML

Semantic HTML is the foundation of accessibility. Using the right elements gives screen readers and assistive technologies built-in meaning.

〈 Landmarks (HTML5 semantic elements)
<header>       <!-- Site header / banner -->
<nav>          <!-- Navigation -->
<main>         <!-- Primary content -->
<article>     <!-- Self-contained content -->
    <h1>Title</h1>
    <p>Content</p>
</article>
<aside>       <!-- Complementary content -->
<footer>       <!-- Site footer -->
💡 Rule of thumb: Use HTML elements for their intended purpose. A <button> is automatically focusable and announces as "button". A <div> with an onclick does neither.

3. ARIA (Accessible Rich Internet Applications)

ARIA provides attributes that add accessibility information when HTML semantics aren't enough. Use it to enhance, not override, native semantics.

〈 ARIA Roles, Properties & States
<!-- Role -- tells what an element is -->
<div role="button" tabindex="0">Click</div>

<!-- Property -- gives extra meaning -->
<nav aria-label="Main navigation">

<!-- State -- dynamic condition -->
<button aria-expanded="false">Menu</button>

<!-- Live region -- announces changes -->
<div aria-live="polite">
▶ ARIA Live Demo

Click the button to see a live region in action:

A screen reader will announce the message automatically because of aria-live="polite".

⚠️ First rule of ARIA: Don't use ARIA if you can use a native HTML element. <button> already has role="button", keyboard handling, and focus management built in. Adding ARIA to a <div> requires manual implementation of all of these.

4. Keyboard Navigation

Many users navigate entirely with the keyboard. Every interactive element must be reachable and operable via keyboard.

〈 Tabindex & Focus
<!-- Natural tab order (recommended) -->
<button>First</button>
<button>Second</button>
<a href="#">Link</a>

<!-- tabindex values -->
tabindex="0"   <!-- Focusable in natural order -->
tabindex="-1"  <!-- Programmatically focusable -->
tabindex="5"   <!-- AVOID! Use natural order -->

<!-- Skip link for keyboard users -->
<a href="#main-content"
 class="skip-link">Skip to main content
</a>

5. Color & Contrast

Color should never be the only way to convey information. Text and interactive elements must have sufficient contrast against their backgrounds.

💡 WCAG minimums: Normal text needs a 4.5:1 ratio (AA) or 7:1 (AAA). Large text (18px+ bold or 24px+ regular) needs 3:1 (AA) or 4.5:1 (AAA). UI components and icons need 3:1.

6. Screen Readers

Screen readers convert text to speech or braille. Proper HTML and ARIA ensure screen reader users get the same information as sighted users.

〈 Alt Text & Descriptions
<!-- Decorative: empty alt -->
<img src="divider.png" alt="">

<!-- Informative: describe content -->
<img src="chart.png"
    alt="Bar chart showing"
    "revenue up 25% in 2025">

<!-- Complex: reference aria-describedby -->
<img src="diagram.svg"
    aria-describedby="desc1">
<p id="desc1">Full description...</p>
▶ Screen Reader Demo

Toggle the hidden text to see what screen reader users hear:

Image with alt text
Screen reader announces: "A random landscape photo from Lorem Picsum"
💡 Alt text tips: Describe the content and function of the image, not the image itself. Instead of "blue button", say "Search". Decorative images get alt="" (empty) so screen readers ignore them.

7. Accessible Forms

Forms are one of the most common accessibility failures. Every input needs a label, clear instructions, and understandable error messages.

〈 Accessible Form Pattern
<form>
<fieldset>
    <legend>Personal Info</legend>

    <label for="name">
    Name <span aria-hidden="true">*</span></label>
    <input type="text" id="name"
        aria-required="true"
        aria-describedby="nameHint">
    <p id="nameHint">Enter full name</p>

    <button type="submit">Submit</button>
</fieldset>
</form>

8. Focus Management

Visible focus indicators help keyboard users track their location on the page. Never remove the focus outline without providing an alternative.

〈 Focus Styles
/* GOOD: visible focus ring */
button:focus-visible {
  outline: 3px solid #667eea;
  outline-offset: 2px;
}

/* BAD: hidden focus */
button:focus {
  outline: none; /* Don't do this! */
}

/* Focus trap for modals */
.modal:focus {
  outline: none;
}
// JS: trap focus within modal
▶ Focus Demo

💡 :focus-visible vs :focus: Use :focus-visible to show the focus ring only when the user navigates via keyboard (not on mouse click). This gives a cleaner look for mouse users while keeping accessibility for keyboard users.

9. Testing Tools & Checklist

Accessibility testing should be done throughout development, not at the end. Here are tools and a quick checklist to get started.

Lighthouse

Built into Chrome DevTools. Generates an accessibility score and lists specific issues with fix guidance.

axe DevTools

Browser extension that finds and explains accessibility issues. Free tier available.

WAVE

Visual overlay that shows accessibility issues directly on the page. Also checks contrast.

Keyboard Testing

Unplug your mouse. Navigate using only Tab, Shift+Tab, Enter, Escape, and arrow keys.

💡 Quick checklist: All images have alt text ✓ Headings are in order ✓ Forms have labels ✓ Color contrast passes 4.5:1 ✓ Keyboard navigation works ✓ Page has a skip link ✓ ARIA is used correctly ✓ Tested with a screen reader ✓
